# Aztec Private Voting — Context for Claude Code

## What this is

Private ballot infrastructure for DAOs. Four React components + Aztec Noir smart contracts. Members vote privately — only the final tally is revealed.

**The research contribution:** The receipt UX — designing for "trust without transparency." After submitting a private ballot, the voter gets a cryptographic nullifier. In every existing system this is a hex string. We're making it human-readable and trustworthy.

Full design rationale: `docs/receipt-design.md` — **read this before touching any receipt-related code.**

## Key design decisions

**"Vote fingerprint" not "nullifier"** — the UX term for the nullifier. Never use "nullifier" in user-facing copy.

**Receipt does not contain the vote choice** — this is intentional and the point. The receipt proves a vote was counted without revealing what was voted.

**Private/public split in cast_vote** — the private function `cast_vote` is witness-only; it enqueues `record_vote` (public) which does all state reads and writes. Do not put state reads in the private half.

**Double-vote guard = private single-use claim per wallet** — `vote_claims` (Owned<SingleUseClaim>) consumed inside `cast_vote`; its protocol nullifier derives from the caller's keys in the private kernel, so observers cannot link wallets to ballots. The receipt id is a separate, client-generated random field. Known limitation (L1): per-ballot choices are public args of `record_vote` (anonymous but plaintext) — a voter who shares their fingerprint reveals their choice. Encrypted tally is roadmap M2.
