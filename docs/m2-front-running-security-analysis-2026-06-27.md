_Author: Jony Bursztyn · 2026-06-27_

_Scope: `cast_vote_babylon_v2`, M2 secp256k1 ownership proof path_

---

## 1. The open question from M2 §11

The M2 design spec §11 flags:

> _"Front-running within the same block: If an attacker intercepts the signed proof after it is
> submitted but before it lands on-chain, they could potentially replay it in the same block.
> Aztec's private kernel applies the single-use claim before the proof hits the sequencer, so
> ordering within a block is handled, but this is worth confirming against v5 sequencer semantics."_

This document resolves that question and surfaces a related but distinct double-vote risk.

## 2. Front-running analysis

### 2a. `record_vote` is `#[only_self]`

The public half of the vote enqueuing pipeline is decorated `#[only_self]`:

```noir
#[external("public")]
#[only_self]
fn record_vote(vote_choice: u8, eligibility_proof: Field, receipt_id: Field) { ... }
```

`#[only_self]` in Aztec means the function is callable exclusively via `enqueue_self` from within
the contract's own private context. An external account — including a malicious sequencer or
observer — cannot call `record_vote` directly with an observed `holder_nullifier`. This closes the
most obvious front-running vector: the attacker cannot inject a record using a stolen nullifier.

**Result: direct front-run via stolen nullifier is blocked.** ✅

### 2b. Nullifier preimage resistance

M2's nullifier is:

```noir
let holder_nullifier = hash_bytes_as_field(sha256_var(sig, 64));
```

The nullifier is derived from the 64-byte ECDSA signature, not from any publicly available value.
An observer who sees the pending enqueued call's `receipt_id` argument (the nullifier) cannot work
backwards to the signature (`sha256` preimage resistance), and cannot produce a different signature
that hashes to the same nullifier (second-preimage resistance). They cannot construct a valid
`cast_vote_babylon_v2` proof without the holder's secp256k1 private key.

**Result: nullifier cannot be pre-computed or stolen by an observer.** ✅

### 2c. Transaction replay

In Aztec's private execution model, a private transaction proof is generated by the PXE and
submitted to the sequencer. The private kernel circuit binds the proof to the execution context
(call frame, nonce, contract address). An exact-byte replay of a submitted transaction would be
rejected by the sequencer via standard transaction deduplication — the same reason Ethereum
transactions cannot be replayed verbatim once included.

Additionally, even if an attacker obtained the full proof bytes and re-submitted them, the outcome
would be:

1. The same `holder_nullifier` would be produced.
2. `receipts.at(holder_nullifier)` would already be `true` (if the original tx landed first), so
   `record_vote` would revert with "receipt already used".
3. If the replay somehow landed first, the original transaction would be the one that reverts —
   which is the same griefing scenario as the F2 open-path analysis; the atomicity guarantee
   means the victim can retry.

**Result: transaction replay is not exploitable for a net-new vote.** ✅

### 2d. Sequencer visibility of `vote_choice`

The enqueued `record_vote(vote_choice, 1, holder_nullifier)` call has `vote_choice` as a public
argument. A malicious sequencer can read this value before the transaction is finalized.
This reveals the voter's choice to the sequencer — a known limitation documented as `L1 privacy`
in the contract and on the M2 roadmap. It is not a front-running risk (the sequencer cannot use
this knowledge to cast a vote on behalf of the holder) but it is a remaining privacy gap for the
Babylon integration. **M3 (hidden tallies) resolves this.**

**Result: `vote_choice` is visible to the sequencer; no front-running from this; privacy gap
documented and on roadmap.** ⚠️ (known limitation, not new)

## 3. New finding: RFC 6979 determinism dependency

### 3a. Background

Unlike the open vote path (`cast_vote`), `cast_vote_babylon_v2` does **not** call:

```noir
self.storage.vote_claims.at(self.msg_sender()).claim();
```

The open path's `vote_claims` mechanism produces a protocol-level Aztec nullifier that is:
- Derived from the caller's Aztec secret keys
- Single-use per wallet address, enforced by the private kernel
- Completely independent of the vote's application-layer logic

M2 omits this because Cosmos voters are identified by their secp256k1 keys, not by an Aztec
address. The Aztec `msg_sender()` would be whatever Aztec account submitted the transaction —
potentially a shared relayer — and is not meaningful for Babylon-mode eligibility.

Instead, M2's single-use guarantee is **entirely** delegated to the `receipts[holder_nullifier]`
map in `record_vote`. One vote per holder is enforced if and only if the same holder always
produces the same nullifier for the same vote.

### 3b. The nullifier is `hash(sha256(sig))`

Under RFC 6979, ECDSA signature generation is deterministic:

```
k = HMAC_DRBG(sha256, private_key, message)
(r, s) = ecdsa_sign(k, private_key, message)
```

Given the same `(private_key, message)`, RFC 6979 always produces the same `(r, s)`, and
therefore the same `holder_nullifier`. This means:

- Same Cosmos key + same vote = same sig = same nullifier → double-vote blocked ✅
- Different vote (different `title_hash || root` challenge) = different sig = different
  nullifier → voting in each vote is independent ✅

### 3c. The gap: RFC 6979 is not mandated by the ECDSA spec

Standard ECDSA (SEC 1 / ANSI X9.62) permits any unpredictable nonce `k` — not just the
deterministic one from RFC 6979. A wallet that generates a random `k` on every signing call will
produce a different `(r, s)` each time for the same `(private_key, message)`.

If a voter uses such a wallet, or manually randomises their signing nonce, they can produce
distinct valid signatures, each generating a distinct `holder_nullifier`, and submit multiple
votes in the same election.

**In practice:** EVM wallets (MetaMask, Ledger via MetaMask, WalletConnect) use RFC 6979 for
`personal_sign` / `eth_sign` — this is enforced by the EIP-191 / Ethereum spec. Since ADR-036
selects EIP-191 Path C as the signing method, the current implementation is safe for EVM wallets.

**However:** If the system is extended to support Cosmos-native signing (Keplr, Ledger Cosmos
app), the Cosmos SDK uses RFC 6979 for secp256k1 by default — but this is not formally
contractual, and some hardware wallet firmware implementations have been observed to add
additional entropy. **This dependency should be named and documented.**

### 3d. Severity

| Scenario | Exploitable? | Notes |
|---|---|---|
| MetaMask EIP-191 `personal_sign` (current, ADR-036 Path C) | No | MetaMask uses RFC 6979 |
| Cosmos Keplr browser extension | Probably no | Cosmos SDK uses RFC 6979 by default |
| Hardware wallet with randomised k | Yes | Rare; bounded by Merkle eligibility (one claim per snapshot leaf) |
| Deliberate non-RFC-6979 signing tool | Yes | Requires attacker to hold the private key — not an external attacker, but a malicious voter |

**Net severity:** LOW for current EIP-191 deployment (MetaMask + RFC 6979 confirmed). Medium
concern for Cosmos wallet extension (Keplr path, ADR-036 Path A — documented as a named extension
for future work). The risk is bounded: an attacker must own the secp256k1 private key and use
non-standard signing. They cannot inflate their vote weight (the Merkle leaf pins their balance).
They can, at worst, submit multiple valid-looking votes with different nullifiers — all recording
the same `vote_choice` if they want the votes to land. Each subsequent vote consumes Aztec gas.

### 3e. Mitigation options

**Option 1 — Document as named dependency (recommended for current scope):**  
Add a named trust assumption to `cast_vote_babylon_v2` and to the M2 design doc: "Correct
single-use enforcement assumes the signing wallet implements RFC 6979 deterministic nonce
generation (standard for all EIP-191 EVM wallets and Cosmos SDK). Non-RFC-6979 signing allows
multiple votes from the same key pair." Update GRANT.md to note this is out of M2 scope and
addressed in M3/production hardening.

**Option 2 — Add protocol-level claim per secp256k1 pubkey:**  
Store `hash(pubkey_x || pubkey_y)` in a dedicated `Owned<SingleUseClaim>` storage slot keyed by
pubkey hash. This adds ~1 storage slot and is independent of the Aztec address. It eliminates the
RFC 6979 dependency entirely.

Downside: requires a snapshot-level pubkey hash for each holder (currently the snapshot only
stores `hash160` in M2 leaf format). The snapshot generator must be extended.

**Option 3 — Derive nullifier from commitment, not sig:**  
Use `hash(title_hash || hash160 || balance)` as the nullifier instead of `hash(sha256(sig))`.
This is deterministic and independent of signing nonce. Downside: the nullifier is computable
from public data (anyone with the snapshot can enumerate holders and compute nullifiers), re-
introducing the pre-M2 snapshot-enumeration attack. Rejected.

**Recommendation:** Option 1 for the current grant demo and CHI paper scope. Option 2 before
any production deployment handling real governance stakes.

## 4. Summary table

| Risk | Status | Mechanism |
|---|---|---|
| External caller directly invoking `record_vote` with stolen nullifier | ✅ Closed | `#[only_self]` |
| Attacker forges a valid M2 proof without private key | ✅ Closed | secp256k1 + in-circuit signature check |
| Attacker pre-computes nullifier from snapshot data | ✅ Closed | Nullifier = `hash(sha256(sig))`; sig is private |
| Transaction byte replay | ✅ Closed | Aztec tx deduplication + receipt atomicity |
| Sequencer learns `vote_choice` | ⚠️ Known | Public arg in `record_vote`; M3 scope |
| Double-vote with non-RFC-6979 wallet | ⚠️ Low (named) | RFC 6979 assumed; EVM wallets confirmed; Keplr presumed |
