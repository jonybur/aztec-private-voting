# The Proof-of-Inclusion UX Pattern

_Author: Jony Bursztyn · 2026-06-22_  
_Related: [`docs/receipt-design.md`](receipt-design.md) (design rationale), [`docs/receipt.md`](receipt.md) (threat model)_

---

## The problem class

A class of systems has the following structure:

1. A user submits private content to a public or semi-public system.
2. The system records that a submission occurred, but is not supposed to reveal the content.
3. The user may later need to confirm their submission was received and processed.

Examples beyond voting:
- Anonymous whistleblower submissions (the system knows a report was filed; neither the reporter nor the system should be able to prove what was reported to a third party)
- Sealed bids in an on-chain auction (the bidder needs to confirm their bid was included before reveal)
- Private asset deposits (proof of deposit without proof of amount)
- Blind peer review assignments (a reviewer can confirm they were assigned; the system cannot prove to an editor which reviewer it was)

In each case, the user needs a **receipt** that satisfies two requirements simultaneously:

- **Inclusion proof**: the receipt proves the submission was received and processed
- **Content blindness**: the receipt reveals nothing about what was submitted

These requirements look contradictory. A receipt that says "your submission was processed" without any content identifier gives no way to check whether it refers to _your_ submission or anyone's. A receipt that contains a content-linked identifier (a hash of the content, a transaction hash that encodes the content) reveals the content to anyone who can obtain the receipt.

The tension is not new — it is a variant of the receipt-freeness problem in cryptographic voting theory (Benaloh and Tuinstra, 1994). But the protocol-level solution (using zero-knowledge proofs to decouple the submission event from the submission content) has not had a corresponding UX treatment in the design literature.

This document names the pattern, states the design constraints, and shows the implementation in Aztec Private Voting as a worked example.

---

## The pattern

**Name:** Proof-of-Inclusion UX Pattern (PIUP)

**Core idea:** Give the user an opaque surrogate identity for their submission. The surrogate:

- Is random (not derived from the user, the content, or any observable system state)
- Is recorded by the system as a key into the "processed" set
- Is given to the user as their receipt
- Can be verified against the "processed" set by anyone who holds it — but holding it reveals nothing about the content

Verification becomes: "is this key in the set?" — not "what is this key about?"

The pattern has three invariants:

1. **Surrogate independence**: the surrogate must not be derivable from the content, the user's identity, or any public state. If it is, an observer can predict or reverse-engineer whether a given user submitted, or what they submitted.

2. **Surrogate privacy in transit**: because the surrogate travels with the content during submission (in whatever system record is created), an observer who can read the submission record and then sees the voter's surrogate can link them. The surrogate must be treated as private until the submission is definitionally public (vote is closed, auction is revealed, etc.).

3. **Minimal receipt content**: the receipt artifact given to the user should contain only what is needed to verify inclusion. Nothing more. Every additional field is a potential content leak or identity leak.

## Implementation in Aztec Private Voting

### The surrogate

The vote fingerprint — called `receipt_id` at the protocol layer — is generated by:

```typescript
// packages/react/src/aztec/receipt-id.ts
export async function generateReceiptId(): Promise<bigint> {
  const { Fr } = await import('@aztec/aztec.js');
  return Fr.random().toBigInt();
}
```

`Fr.random()` produces a random 254-bit field element. It is not derived from the wallet, the vote id, the choice, or any other observable. This satisfies Invariant 1.

### The protocol record

The contract records the surrogate without the content:

```noir
// main.nr — record_vote (public)
let already_used = self.storage.receipts.at(receipt_id).read();
assert(already_used == false, "receipt already used");
self.storage.receipts.at(receipt_id).write(true);
```

The `receipts` map is keyed by `receipt_id` and stores a boolean. The vote choice is recorded separately in the tally (also as a boolean count). Nothing in the stored state links `receipt_id` to `vote_choice`.

### The transit caveat (L1 privacy gap)

The Aztec contract structure has a private entrypoint (`cast_vote`, which generates the ZK proof) that enqueues a public call (`record_vote`). The public call carries both `vote_choice` and `receipt_id` as arguments. These are public execution data — visible to observers of the chain.

This is the one place where the PIUP fails to fully satisfy Invariant 2 without cooperation from the voter. An observer watching public execution can build a map: `receipt_id → vote_choice`. A voter who later reveals their `receipt_id` (for example, to prove they voted) lets a knowledgeable observer look up that receipt_id in the execution record and read the choice.

The receipt UI addresses this with explicit copy: the receipt explains that the fingerprint "proves your vote was counted without revealing how you voted" — but the threat model documentation (see `receipt.md` section (e)) adds the crucial caveat that this holds only if the fingerprint is kept private until the tally closes. After the vote finalizes, the tally is public by design, and the L1 privacy gap no longer has practical consequences.

This is a real limitation of the current design. Closing it fully would require encrypting the tally (so `vote_choice` never appears in public execution) — a roadmap item. The PIUP is fully realized only when the surrogate is never co-located with the content in any observable record. The current implementation is an approximation that holds under a weaker assumption: the voter keeps the fingerprint private until the content is public.

### The naming decision

Calling the surrogate a "fingerprint" rather than "nullifier", "receipt ID", or "token" is the smallest HCI intervention in the implementation, and arguably the most important one.

The naming goal is: the user should immediately understand what the artifact is for (uniquely identifies their vote), what to do with it (save it, don't share it until the vote closes), and what it does not contain (their choice). The word "fingerprint" activates this model faster than any technically correct term — at the cost of a slight metaphor imprecision (a fingerprint identifies a person; a vote fingerprint specifically does not identify the voter). The metaphor is close enough that users calibrate on "save this, use it later to check" rather than requiring an explanation of the cryptographic mechanism.

This follows Whitten and Tygar's "Why Johnny Can't Encrypt" (1999) principle that technically correct feedback that violates users' mental models is effectively no feedback. Users shown "nullifier" in usability sessions of comparable systems misread it as "the vote was nullified." The word choice is not cosmetic.

## Generalisation: design checklist for the PIUP

For any system that needs to implement the PIUP:

| Check | Implementation |
|---|---|
| The surrogate is random (not derived from user or content) | `crypto.getRandomValues()`, ZK-random, etc. |
| The surrogate is recorded, not the content | Backend stores `{surrogateKey: seen=true}`, not `{surrogateKey: content}` |
| The receipt contains only the surrogate + metadata (no content, no identity) | Audit the receipt schema before shipping |
| The surrogate is never in the same observable record as the content | Protocol-level constraint; if not achievable, add explicit copy about temporal privacy |
| The user is told: save this, don't share it until [event], use it then to verify | The receipt UI's primary job |
| The verification is public and content-blind | `isInSet(surrogate) -> bool`, not `getContentFor(surrogate)` |
| The surrogate has a user-facing name that activates the right mental model | Not "token", "ID", "hash", "nullifier" — something that means "unique to me, not about me" |

## Why this matters for the HCI/ZK intersection

The majority of zero-knowledge UX research focuses on the prover side: how do you explain to a user what a ZK proof is, and how do you make them trust it? (Relevant: Cai et al. "zkUI", CHI 2024 workshop; Chung et al. "Privacy in Web3", CHI 2023.)

The PIUP addresses a different problem: **not how to explain the proof to the user, but how to give the user a usable artifact from the proof system without that artifact being a leak vector**. The user does not need to understand how the fingerprint was generated. They need to understand what to do with it.

This is closer to the security usability tradition (Cranor and Garfinkel, "Security and Usability", 2005; Ruoti et al. "Why Johnny Still Can't Encrypt", SOUPS 2015) than to the ZK explainability tradition. The design problem is not comprehension of a cryptographic mechanism; it is the design of an artifact whose _affordances_ match the _correct_ interaction pattern (save, don't share prematurely, verify later) without requiring the user to understand why.

The contribution of Aztec Private Voting to this space is a worked implementation of the PIUP in a production ZK system, with specific evidence for the naming decision (fingerprint vs. nullifier), the persistence decision (download as primary action), and the temporal disclosure problem (the L1 privacy gap). None of these decisions appear in the existing ZK/DAO voting literature, which focuses on cryptographic properties and largely skips the UX artifact layer.

## Open questions for future work

1. **Can the PIUP be implemented without the temporal privacy caveat?** Yes, if the content is encrypted at submission and only decrypted after a defined event (vote close, auction reveal). In that case, the surrogate is never co-located with plaintext content in any public record. Aztec's encrypted notes provide the mechanism; the interaction design question is what the UX around encrypted-tally voting looks like.

2. **What happens when the user loses the surrogate?** The current implementation has no recovery path; the user falls back to trusting the public tally. Key-derived surrogates (where the surrogate can be regenerated from the user's wallet seed) offer recovery at the cost of predictability — a coercer with the wallet seed can also regenerate the surrogate. The tradeoff is unstudied in the ZK context.

3. **Does the "fingerprint" naming hold up in comprehension testing with non-technical users?** The current evidence is informal (internal review, comparable-system usability sessions). A controlled study comparing "fingerprint", "token", "receipt code", and the hex value alone would strengthen the naming decision or reveal a better option.

4. **Does the verification affordance actually change behaviour?** The receipt presents a collapsed "How to verify" section. If most users never expand it, let alone come back to verify, the functional value of the proof-of-inclusion is largely psychological. Whether psychological trust is sufficient (or whether it creates a false sense of security when the verification mechanism is never actually used) is an open design and ethics question.

## Relationship to existing work

The closest prior work is the UX research around Helios (Adida et al., 2008) and STAR-Vote (Bell et al., 2013), both of which grappled with making cryptographic audit trails comprehensible to voters. Helios showed that verifiable election systems can be usable by novices when the verification steps are concrete and immediate; STAR-Vote added the paper ballot as an anchor for the digital receipt. Neither system is ZK-based, and neither addresses the problem of the receipt artifact being a coercion tool (because both systems reveal the vote choice in the receipt for post-hoc verification).

The PIUP's contribution is specifically the **combination of verifiability with content-blindness in the receipt artifact**. Helios and STAR-Vote achieve verifiability; MACI achieves content-blindness; neither achieves both in the same receipt UX. The Aztec Private Voting receipt achieves both, under the temporal privacy caveat, using the ZK nullifier as the surrogate and a naming decision to make the artifact usable.
