## §3.5 M2 ownership proof ⚠️ INACCURACY — FLAG FOR JONY (not edited)

**Reason not edited:** §3.5 describes the Babylon-compatible governance path (`cast_vote_babylon_v2`, `useVoteBabylonV2`). Per standing direction set 2026-06-22, the Babylon governance path in aztec-private-voting is off-limits for editing.

**Inaccuracy found:**

Paper §3.5 says:
> "The M2 path uses EIP-191-compatible message encoding (implemented in `useM2Signing.ts`) and **combines the signature components into the `sig[64]` format** expected by the in-circuit secp256k1 verifier."

**What the code actually does:**

| Hook | Responsibility |
|---|---|
| `useM2Signing.ts` | Handles EIP-191 message encoding + signature extraction. Returns `M2SigningOutput` with **separate** `sig_r: number[]` and `sig_s: number[]` fields. Does NOT combine. |
| `useVoteBabylonV2.ts` (line 91) | **Combines** `sig_r \|\| sig_s` → `sig[64]` for the circuit: `const sig = [...input.m2Signing.sig_r, ...input.m2Signing.sig_s]` |

**Correct description should be:**
> "The M2 path uses EIP-191-compatible message encoding (implemented in `useM2Signing.ts`); `useVoteBabylonV2` then combines the signature components (`sig_r || sig_s`) into the `sig[64]` format expected by the in-circuit secp256k1 verifier."

**Jony-action:** Fix §3.5 when reviewing the Babylon governance path sections. One-sentence change: attribute `sig[64]` combination to `useVoteBabylonV2.ts`, not `useM2Signing.ts`.

**ADR-036 EIP-191 circuit description (commit 05d83bb):** Otherwise accurate. The EIP-191 prefix, keccak256 hashing of the 60-byte wrapped message, and `ecdsa_secp256k1::verify_signature` call are correctly described in the paper's §3.5 framing (the paper says "EIP-191-compatible message encoding" without spelling out the circuit detail, which is appropriate for a CHI venue).
